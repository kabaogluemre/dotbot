Dotbot fails to read attachments from Confluence and Jira: PDFs, Excel, screenshots, images
Bug

Dotbot struggles to pull and review files attached to Confluence pages, including PDFs, Excel spreadsheets, screenshots, and images. It appears to read inline text from Confluence reasonably well, but fails on attached items. The same issue likely applies to Jira attachments. This causes requirements and mapping documents to be partially or entirely missed even when dotbot ostensibly has access to them.

Bug

One confirmed trigger is restarting dotbot on an existing project: tasks already present in the project get created again, resulting in duplicates rather than resumption. Bug

User E
Repository not found: naming mismatch between business language and DevOps reality
Bug

Dotbot cannot locate the correct repository because business analysts refer to systems by their business names (e.g. "Portal App", "Client App") while Azure DevOps uses entirely different identifiers (e.g. project "Org.Portal", repository "internal-codename"). Brand names add another layer of confusion: "New BrandX E-Commerce" is misleading because it is a brand name but the repo is actually a white-label e-commerce platform. Repo selection prompts are also currently ignored in general (see feature: upfront repo selection).

Bug

User E

Bug

Dotbot uses the JIRA MCP assuming an out-of-the-box field configuration, but many organisations have customised JIRA with non-standard fields. This causes dotbot to create malformed tickets, miss required fields, or fail silently. Reported in session
Longer autonomous tasks show increasing silent failure rate
Bug

Tasks that run longer and require little or no user input during execution are failing at a higher rate than shorter, more interactive tasks. Critically, these failures are silent: dotbot does not surface an error, ask for help, or flag that anything has gone wrong. The combination of long autonomous runs and silent failure makes these problems very difficult to detect and recover from. Dotbot should monitor its own progress during execution and surface stalls, errors, or unexpected states proactively rather than waiting for the user to notice.

User H
Dotbot spawns PowerShell processes that cannot be safely cleaned up
Bug

Dotbot spawns child PowerShell processes to run tasks, but these processes accumulate and cannot be reliably cleaned up without risking killing the main dotbot process. There is no safe teardown mechanism. Dotbot should manage its spawned processes with proper lifecycle control so they can be terminated cleanly without side effects.

User B

Bug

The process continues without the content, producing incomplete outputs silently. Dotbot should perform a pre-check on all linked resources before starting a session, surface any access failures clearly, and hold the process until access is confirmed or the resource is excluded.

User I
code search tool rate limiting handled opaquely: unclear whether dotbot skipped or retried
Bug

Dotbot was observed receiving rate-limit responses from code search tool during a session, despite the organisation not having configured any rate limiting.
